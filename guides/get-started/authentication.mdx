Las llamadas a la API se deben autenticar mediante tu **API Key** en cada petición.

<Warning>
  Tu API Key conlleva muchos privilegios por lo que asegúrate de mantenerla en secreto. No compartas tu API Key ni la expongas nunca en el código del lado del cliente de cualquier sitio web público.
</Warning>

## Obtener tu API Key
Puedes encontrar tu API Key de comercio en tu panel del Backend de Paylands. Pulsando en el icono superior derecha se desplegarán tus credenciales.

## Método de Autenticación
La autenticación de la API se realiza mediante **HTTP Basic Auth**. Provee tu API Key como valor para el campo username (nombre de usuario). No es necesario proveer valor para el campo password (contraseña). Si se setea la cabecera Basic Auth manualmente (como por ejemplo, al hacer una prueba con curl), la API Key debe ir codificada en base64. Si se usa Postman, Insomnia o cualquier otra aplicación similar, estas ya la codifican en base64 automáticamente al enviar la petición.

Si necesitas autenticar vía **Bearer authentication** (como por ejemplo, para una petición cross-origin), usa el modificador `-H "Authorization: Bearer <API Key>"` en lugar de `-u <API Key>:`.

Por ejemplo, si tu API Key es `5d66a96f460f4109b24c2243fb1c43e1`, las siguientes cabeceras serán válidas:

```http
Authorization: Basic NWQ2NmE5NmY0NjBmNDEwOWIyNGMyMjQzZmIxYzQzZTE=
```

```http
Authorization: Bearer 5d66a96f460f4109b24c2243fb1c43e1
```

Todas las peticiones a la API deben realizarse sobre HTTPS. Las peticiones realizadas sobre HTTP serán denegadas; así como las peticiones sin autenticar.

```bash
curl https://api.paylands.com/v1/sandbox/orders \
  -u "YOUR_API_KEY:"
```

## Perfiles API key
La API key tiene asignados varios perfiles que determinan sus permisos. En la Referencia API puedes encontrar información sobre los diferentes perfiles de la API key y cómo obtenerlos.


## Tarjetas de Prueba
Paylands provee un entorno de integración en sandbox, que permite al comercio integrar diferentes operativas. A continuación se muestran las tarjetas de prueba que podrán utilizarse:

| Operativa | Tarjeta |
| :--- | :--- |
| Pago seguro OK | 4761739000091011 |
| Pago seguro KO | 5150796024238164 |
| Pago no seguro OK | 5590337536718399 |
| Pago no seguro + SCA | 5223798193659108 |
| Pago no seguro KO | 5485009949201879 |
| Pago MoTo OK | 5150796024238164 |
| Pago MoTo KO | 5485009949201879 |

El CVV siempre será 123 y se recomienda utilizar como fecha de expiración 12/34. En el caso de indicar otra, deberá ser posterior a la fecha actual y siguiendo el formato mm/aa. El campo del titular de la tarjeta es indiferente, por lo que puede introducirse el valor que se desee.

En el caso de un pago no seguro, a raíz de la entrada de la PSD2, el banco emisor puede requerir intervención del usuario, lo que se conoce como Strong Customer Authentication (SCA), provocando un soft decline en la operación. Esto significa que el usuario debe ser redirigido al 3DS para finalizar el pago. La respuesta del endpoint de pago directo en este caso devolverá un código http 303 y en el cuerpo de la respuesta contendrá una url a la que el comercio deberá redirigir al usuario para que finalice la operación por 3DS.

Las nuevas versiones del 3DS v2 requieren de pasos intermedios que Paylands realiza de forma automática, por lo que el comercio sólo debe tener en cuenta las tarjetas anteriores para poder probar los diferentes tipos de pago con el estado que se desee (SUCCESS/REFUSED).

No obstante, en caso de desearlo, se proporcionan también tarjetas de prueba para poder simular cualquiera de los pasos intermedios en el proceso de cobro de una operación.

### 3DS v1.0
Las tarjetas con 3DS v1 son aquellas que o bien fueron emitidas antes de la entrada de la PSD2, o aquellas en las que el banco no las ha configurado para soportar los nuevos flujos 3DS v2. En estos casos, Paylands redirige al usuario al 3DS v1.0 de su banco para que finalice la operación.

| Resultado | Tarjeta |
| :--- | :--- |
| v1.0 OK | 5245172944073155 12/34 123 |
| v1.0 KO (MSP) | 4024007196210529 12/34 123 |
| v1.0 KO (Init3DS) | 5514214341183643 12/34 123 |
| v1.0 KO (AuthV1) | 5182246146680010 12/34 123 |

### 3DS v2.2 Frictionless
El flujo de pago Frictionless consiste en redirigir al usuario al 3DS, pero donde el banco emisor (o, en este caso, el simulador) decide que no es necesario que el usuario intervenga, provocando que la operación finalice automáticamente con un estado de SUCCESS o REFUSED.

| Resultado | Tarjeta | Información adicional |
| :--- | :--- | :--- |
| Frictionless OK | 5590337536718399 12/34 123 | |
| Frictionless KO (MSP) | 5503460708601080 12/34 123 | |
| Frictionless KO (Init3DS) | 5584518933010195 12/34 123 | |
| Frictionless KO (AuthV2) | 5342125074190670 12/34 123 | |
| Frictionless KO (Init3DS) | 5123450000000008 12/34 123 | Incluye el campo error_details completo, con un código MAC que indica que el pago no debe reintentarse |
| Frictionless KO (Init3DS) | 5111111111111118 12/34 123 | Incluye el campo error_details completo, con un código MAC que permite reintentar el pago transcurrido cierto tiempo |

### 3DS v2.2 Sólo Challenge
El flujo de sólo challenge consiste en redirigir al 3DS, el cual puede dar tres respuestas, dependiendo de la elección del usuario: SUCCESS (OK), ATTEMPT (KO) o REFUSED (KO). El estado de ATTEMPT se utiliza para simular un error en el proceso de conexión con el banco emisor, como por ejemplo, cuando el banco del usuario no tiene disponible el 3DS.

| Resultado | Tarjeta |
| :--- | :--- |
| Solo Challenge OK | 4548817212493017 12/34 123 |
| Solo Challenge KO (MSP) | 5404154444859348 12/34 123 |
| Solo Challenge KO (Init3DS) | 4024007114039562 12/34 123 |
| Solo Challenge KO (AuthV2) | 4916393928615990 12/34 123 |

### 3DS Version v2.2 Fingerprinting
El flujo Fingerprinting consiste en capturar, en una pantalla intermedia del flujo de pago, los datos del dispositivo donde se está realizando el pago. El usuario no deberá realizar ninguna acción, ya que esta pantalla de forma automática capturará los datos del dispositivo y, una vez recopilados, redirigirá al 3DS para finalizar la operación.

| Resultado | Tarjeta |
| :--- | :--- |
| Fingerprinting OK | 4761739001010010 12/34 123 |
| Fingerprinting KO (MSP) | 5110753687112003 12/34 123 |
| Fingerprinting KO (Init3DS) | 4414095707030385 12/34 123 |
| Fingerprinting KO (AuthV2) | 4929745158405456 12/34 123 |

### 3DS v1.0 Fallback
El flujo v1 Fallback consiste en tarjetas 3DS v2 que no admiten el pago con su versión predefinida por lo que se debe procesar el 3DS mediante la versión 1.0. En estos casos, Paylands automáticamente redirige al 3DS de la versión 1.0 pudiendo finalizar la operación con cualquiera de los estados habituales (SUCCESS (OK), ATTEMPT (KO) o REFUSED (KO)).

| Resultado | Tarjeta |
| :--- | :--- |
| Fallback OK | 4761340000000035 12/34 123 |
| Fallback KO (MSP) | 5278269366564578 12/34 123 |
| Fallback KO (Init3DS) | 5407659329489182 12/34 123 |
| Fallback KO (AuthV2) | 5167407669858305 12/34 123 |

### 3DS Version v2.2 Full flow
El flujo de pago Full flow consiste en tarjetas que aceptan 3DS v2.2 donde el flujo de pago pasa por todos los estados intermedios hasta que finaliza. Inicialmente se aplica el fingerprinting y luego se redirige al 3DS donde se podrá seleccionar cualquiera de las respuestas SUCCESS (OK), ATTEMPT (KO) o REFUSED (KO).

| Resultado | Tarjeta |
| :--- | :--- |
| Full flow OK | 4761739000091011 12/34 123 |
| Full flow KO (MSP) | 4716280783940260 12/34 123 |
| Full flow KO (Init3DS) | 5203615481177227 12/34 123 |
| Full flow KO (AuthV2) | 5150796024238164 12/34 123 |

A continuación se indica, para estas tarjetas que simulan los pasos intermedios, en qué parte del flujo se rechaza la operación dependiendo del tipo de la tarjeta.

*   **MSP**: significa que el proceso ha llegado al 3DS pero la autenticación ha sido rechazada.
*   **Init3DS**: significa que el proceso 3DS no ha llegado a inicializarse.
*   **AuthV1 o AuthV2**: significa que el proceso 3DS se ha inicializado, pero la redirección al pago seguro no se ha podido realizar.

### Bizum
En el caso de la operativa Bizum hay dos números que simulan un pago correcto y otro incorrecto. Si se utiliza un número diferente a estos dos, la operación será rechazada con un error indicando que el número de teléfono es incorrecto.

| Resultado | Número de teléfono |
| :--- | :--- |
| Pago OK | +34 700 00 00 00 |
| Pago KO | +34 900 00 00 00 |
