# Atributos

Um canal é sempre composto pelos seguintes atributos:

| NOME DO CAMPO | VALORES DISPONÍVEIS                        | DESCRIÇÃO                                                                                                                  |
| ------------- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| endpoint      |                                            | Endpoint para o qual ou a partir do qual o corpo do pedido é transmitido                                                   |
| action        | tokenize, detokenize                       | Especifica se os dados do cartão estão mascarados ou desmascarados.                                                        |
| direction     | pull, push                                 | Indica se o corpo da carga útil que deve ser processado é enviado para o canal ou obtido a partir do canal.                |
| contentType   | json,xml                                   | Formato do corpo da carga útil do qual os dados devem ser extraídos para o processo de tokenização ou destokenização.      |
| encoding      | utf8, ascii, utf16le, ucs2, base64, latin1 | Codificação do corpo do pedido                                                                                             |
| fieldMapping  |                                            | Indica ao proxy PCI como os dados vão ser extraídos do corpo                                                               |

Um `fieldMapping` é composto pelos seguintes campos:

| NOME        | DESCRIÇÃO                                                                                                                                                                                                                                          |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| referencia  | Opcional. Pode ser utilizado para associar os dados do cartão a uma referência do seu software. Por exemplo, um identificador de reserva.                                                                                                          |
| pan         | Número do cartão ou o token do cartão.                                                                                                                                                                                                             |
| cardHolder  | Nome do titular do cartão.                                                                                                                                                                                                                         |
| expiry      | Data de validade do cartão. Os formatos válidos são MM-YYYY, MM-YY, YYYY-MM, YY-MM, MM/YYYY, YYYY/MM, MM/YY, YY/MM, MMYY. Se a data de validade estiver dividida em dois campos separados, utilize em vez disso expiryMonth e expiryYear.          |
| expiryMonth | Utilizado se a data de validade do cartão estiver dividida em dois campos.                                                                                                                                                                         |
| expiryYear  | Utilizado se a data de validade do cartão estiver dividida em dois campos.                                                                                                                                                                         |
| cardCVV     | Número CVV do cartão                                                                                                                                                                                                                               |

O mapeamento de campos é especificado como atributos XPATH ou JSONPATH dependendo do contentType do corpo. Para compreender melhor estas linguagens de consulta, visite os recursos [XPATH](https://www.w3schools.com/xml/xml_xpath.asp) e [JSONPATH](https://goessner.net/articles/JsonPath/index.html#e2).

<Info>
  Para testar as suas expressões XPATH recomendamos esta [Ferramenta de teste XPATH online](https://www.freeformatter.com/xpath-tester.html). Para testar as suas expressões JSONPATH recomendamos esta [Ferramenta de teste JSONPATH online](https://jsonpath.com/).
</Info>

# Quando usar cada configuração de canal

A principal diferença entre as configurações dos canais são os campos **direção** e **ação**. Para que fique mais claro, vejamos alguns cenários do mundo real:

## PUSH + TOKENIZAÇÃO

![PUSH + TOKENIZATION](/images/docs/proxy-pci/img/docs/proxy-pci/push_and_tokenization.png)

É um PMS e um channel manager envia novas reservas para o seu software de backend (por exemplo: SOAP API). Estas reservas incluem dados sensíveis de cartões de pagamento que não deseja armazenar diretamente na sua base de dados, mas sim delegá-los no cofre seguro PCI do PROXY.

Neste caso teria de criar o channel do tipo **PUSH + TOKENIZATION**. No campo *endpoint* adicionará o seu ponto de entrada da API SOAP (por exemplo: `http://somehost.com/xmlsoapinterface`).

Depois de ter criado o channel utilizando a API REST do PROXY PCI, pode atualizar a configuração do channel manager e substituir o seu endpoint pelo endpoint de invocação do channel do proxy pci (p. ex: `https://pci-proxy-api.paynopain.com/production/channel/1df28290-f3cd-411f-9f5f-6b844fe6aaa4/invoke`).

Deste modo, o channel manager invocará o PROXY PCI, que fará a sua magia e finalmente reencaminhará as reservas para o seu endpoint configurado, tendo substituído todos os panes do cartão por uma versão de token do mesmo. Este token pode ser armazenado por si e utilizado para emitir um pagamento ou reencaminhá-lo para outro terceiro.

## PULL + TOKENIZAÇÃO

![PULL + TOKENIZATION](/images/docs/proxy-pci/img/docs/proxy-pci/pull_and_tokenization.png)

É um PMS, mas ainda precisa das reservas de um channel manager. Porém, neste caso, o channel manager não lhe envia os dados. É você quem inicia um pedido ao channel manager e depois o channel manager responde com as últimas reservas.

Neste caso precisa de um channel do tipo **PULL + TOKENIZATION** onde o atributo endpoint do channel é o endpoint API do channel manager. Pode então iniciar o seu pedido ao endpoint de invocação do channel, e o PROXY PCI obterá as novas reservas do endpoint da API do channel manager, fará a sua magia, e responderá com as novas reservas.

<Info>
  O método de invocação do canal PROXY PCI reencaminha todas as QUERY e HTTP HEADERS para o ponto final configurado.
</Info>

## PUSH + DETOKENIZATION

![PUSH + DETOKENIZATION](/images/docs/proxy-pci/img/docs/proxy-pci/push_and_detokenization.png)

Neste caso queremos enviar os dados recolhidos do nosso cartão pan para um PAYMENT GATEWAY de terceiros para processar o pagamento. O PAYMENT GATEWAY oferece um endpoint para isso onde é necessário enviar os dados do cartão no seu formato JSON especificado. Neste cenário precisaremos de um channel do tipo **PUSH + DETOKENIZATION**. O endpoint do channel será o endpoint da API PAYMENT GATEWAY. Enviará os detalhes do cartão, exceto o pan, para o endpoint de invocação do channel PROXY PCI e o PROXY PCI substituirá o token do cartão pelo pan real do cartão.

## PULL + DETOKENIZATION

![PULL + DETOKENIZATION](/images/docs/proxy-pci/img/docs/proxy-pci/pull_and_detokenization.png)

Neste caso de uso final é um channel manager e recolheu os detalhes do cartão de um futuro hóspede utilizando a biblioteca do lado do cliente javascript. Portanto, agora um sistema PMS compatível com PCI quer descarregar os detalhes completos do cartão do seu channel manager. Neste caso, simplesmente configuraria um channel do tipo **PULL + DETOKENIZATION**. No atributo de endpoint especificará o seu endpoint de API e fornecerá o endpoint de invocação PROXY PCI resultante ao sistema PMS, que iniciará o pedido. Nós reencaminharemos o pedido para a sua API e você responderá com a reserva. Substituiremos o token do cartão pelo pan real e responderemos ao PMS.

<Warning>
  Quando se utiliza a ação DETOKENIZATION, o PROXY PCI só substituirá o token e o cvv do cartão pelos seus valores reais. Todos os outros campos têm de ser fornecidos no payload tal como estão. Pode armazenar com segurança a data de validade e o titular do cartão por si próprio.

  O CVV só será substituído se existir no sistema da Paylands. Isto é, se o cartão tiver sido utilizado num pagamento ou se os seus detalhes tiverem sido consultados (incluindo o CVV), o CVV foi consumido e, por requisito PCI, este é eliminado do sistema e, portanto, não será substituído no payload de entrada.
</Warning>

# Exemplos

Para sua melhor compreensão preparámos alguns channels para que os invoque. Há um endpoint de tokenização JSON PUSH e um exemplo de destokenização XML PUSH para que o possa testar.

## Exemplo JSON

```
{  "bookings": [    {      "id": "BK-213123",      "room_category": "VIP",      "facility_id": 1,      "payment_data": {        "card": {          "pan": "4442330588722461",          "cvv": "123",          "expiry": "12/21",          "holder": "John Doe"        }      }    },    {      "id": "BK-3213123",      "room_category": "VIP",      "facility_id": 1,      "payment_data": {        "card": {          "pan": "4701731341611394",          "expiry": "06-2025",          "holder": "Patma Patill"        }      }    }  ]}
```

## MAPEAMENTO DE CAMPOS

| NOME DO CAMPO | VALOR                                     |
| ------------- | ----------------------------------------- |
| reference     | $.bookings\[\*].id                        |
| cardHolder    | $.bookings\[\*].payment\_data.card.holder |
| expiry        | $.bookings\[\*].payment\_data.card.expiry |
| pan           | $.bookings\[\*].payment\_data.card.pan    |
| cardCVV       | $.bookings\[\*].payment\_data.card.cvv    |

## EXEMPLO DE CANAL EM FUNCIONAMENTO

Configurámos um canal com este mapeamento de campos exato. Esta é a configuração exata do canal:

```
{  "action": "tokenize",  "direction": "push",  "endpoint": "https://pciproxypnp.free.beeceptor.com/push/json/tokenize",  "contentType": "json",  "fieldMapping": {    "reference": "$.bookings[*].id",    "cardHolder": "$.bookings[*].payment_data.card.holder",    "expiry": "$.bookings[*].payment_data.card.expiry",    "pan": "$.bookings[*].payment_data.card.pan",    "cardCVV": "$.bookings[*].payment_data.card.cvv"  },  "encoding": "utf8"}
```

Isto significa que se for invocado, o *body* transmitido será analisado e os campos mapeados serão extraídos e guardados. Em seguida, o PAN é substituído por uma versão token do mesmo. Finalmente, o novo *body* é reencaminhado para o endpoint fornecido, incluindo todos os cabeçalhos e parâmetros de consulta originais que foram enviados para o endpoint do channel.

Aceda a [https://beeceptor.com/console/pciproxypnp](https://beeceptor.com/console/pciproxypnp) e deixe a página aberta. Em seguida, execute o seguinte comando `curl`:

```
curl -X POST \
  https://pci-proxy-api.paynopain.com/sandbox/channel/1df28290-f3cd-411f-9f5f-6b844fe6aaa4/invoke \
  -H 'Content-Type: application/json' \
  -H 'SomeCustomHeader: SayHello!' \
  --data '{"bookings":[{"id":"BK-213123","room_category":"VIP","facility_id":1,"payment_data":{"card":{"pan":"4442330588722461","cvv":"222","expiry":"12/28","holder":"John Doe"}}},{"id":"BK-3213123","room_category":"VIP","facility_id":1,"payment_data":{"card":{"pan":"4701731341611394","cvv":"333", "expiry":"06-2025","holder":"Patma Patill"}}}]}'
```

Depois de executar o comando deverá ver o pedido na consola do beeceptor. Sinta-se à vontade para inspecionar o corpo recebido e verificar o campo pan do objeto cartão.

### Namespaces

<Warning>
  Compreender como funcionam xpath e namespaces é crucial para o mapeamento de campos XML.
</Warning>

**Espaços de nomes com prefixos**

Um espaço de nomes é composto por um nome e uma especificação URI. O âmbito da correspondência prefixo-espaço de nomes é o do elemento no qual se produz a declaração do espaço de nomes, bem como o de todos os seus filhos.

```
<bk:bookstore xmlns:bk="urn:xmlns:25hoursaday-com:bookstore"> <bk:book>    <bk:title>Lord of the Rings</bk:title>    <bk:author>J.R.R. Tolkien</bk:author>    <inv:inventory status="in-stock" isbn="0345340426"        xmlns:inv="urn:xmlns:25hoursaday-com:inventory-tracking" /> </bk:book></bk:bookstore>
```

Exemplo: `//bk:author`

**Espaços de nomes não declarados**

Se um elemento tiver o espaço de nomes em branco (xmlns="") então é um espaço de nomes não declarado e qualquer elemento filho sem um prefixo de espaço de nomes está associado a esse espaço de nomes não declarado.

```
<bookstore xmlns="urn:xmlns:25hoursaday-com:bookstore"> <book xmlns="">    <title>Lord of the Rings</title>    <author>J.R.R. Tolkien</author>    <inv:inventory status="in-stock" isbn="0345340426"        xmlns:inv="urn:xmlns:25hoursaday-com:inventory-tracking" /> </book></bookstore>
```

Exemplo: `//title`

### Mapeamento de campos

| NOME DO CAMPO | VALOR                         |
| ------------- | ----------------------------- |
| reference     | //ota:UniqueID/@ID            |
| cardHolder    | //ota:CardHolderName          |
| expiry        | //ota:PaymentCard/@ExpireDate |
| pan           | //ota:PaymentCard/@CardNumber |
| cardCVV       | //ota:PaymentCard/@CardCVV    |

Configurámos um canal com este mapeamento de campos exato. Esta é a configuração exata do canal:

```
{    "action": "detokenize",    "direction": "push",    "endpoint": "https://pciproxypnp.free.beeceptor.com/push/xml/detokenize",    "contentType": "xml",    "fieldMapping": {        "reference": "//ota:UniqueID/@ID",        "cardHolder": "//ota:CardHolderName",        "expiry": "//ota:PaymentCard/@ExpireDate",        "pan": "//ota:PaymentCard/@CardNumber",        "cardCVV": "//ota:PaymentCard/@CardCVV"    },    "encoding": "utf8"}
```

Isto significa que se for invocado, o corpo transmitido será analisado e os campos mapeados serão extraídos. Em seguida, o token é substituído pelo PAN real do cartão.
