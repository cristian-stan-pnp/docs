Quando falamos de **tokenização**, estamos nos referindo à ação de **salvar um método de pagamento para uso futuro**. Ao salvar o método de pagamento, é gerado um token que permitirá emitir cobranças ao cliente sem necessidade de solicitar novamente os dados.

A tokenização protege os dados confidenciais através de um processo de substituição dos dados por um equivalente não confidencial, conhecido como token. O token não tem significado ou valor externo. É uma referência, ou identificador, que através de um sistema de tokenização, é mapeado de volta aos dados confidenciais. Este processo coleta de forma segura informações confidenciais do cartão de crédito e evita o roubo de dados.

<Info>
  Embora normalmente o chamemos de "token" porque "tokenizamos" o cartão, estamos nos referindo ao UUID do cartão. O campo "token" de um cartão é na verdade um valor calculado a partir do número do cartão, de modo que dois cartões com o mesmo [PAN](/guides/resources/glossary#pan) têm UUIDs diferentes mas o mesmo token. Este token serve apenas para saber se dois cartões são iguais. Sempre que queremos utilizar um cartão é necessário indicar seu UUID.
</Info>

## Como salvar um cartão

Podemos armazenar um cartão no Paylands usando um dos seguintes métodos:

### Realizando um pagamento

Podemos tokenizar qualquer cartão ao realizar um primeiro pagamento com ele. O método habitual para comerciantes que não são [PCI](/guides/resources/glossary#pci-dss) é utilizar a [**integração simples via formulário de pagamento**](/guides/accept-payments/web-integration), embora também seja possível iniciar o formulário de pagamento através de:

* [Links de pagamento](/guides/accept-payments/link-payments)
* [Pagamentos Moto](/guides/accept-payments/moto-payments)
* [Plugins E-commerce](/guides/plugins/ecommerce).

Na notificação recebida após o pagamento é incluído um objeto da `ordem` processada. Dentro da transação existe também o objeto `source` que contém o `token` do cartão utilizado no pagamento. Este token ficará a partir de agora vinculado ao cliente, de modo que em novas operações os pagamentos poderão ser confirmados para este usuário sem necessidade de solicitar novamente o cartão.

Estes métodos criam uma ordem com transação [3DS](/guides/resources/glossary#3ds-2) que gera um token com [SCA](/guides/resources/glossary#sca) permitindo que o comerciante envie:

* [MITs](/guides/resources/glossary#mit)
* Cobranças pontuais (incrementais)
* Cobranças recorrentes (assinaturas)

### Integrando o SDK de Tokenização

No caso de estar construindo uma plataforma, web ou mobile, onde se deseja que o usuário possa salvar seu cartão (como Netflix, Amazon) para posteriormente emitir cobranças seja de compras ou de assinaturas, dispomos de uma biblioteca [**SDK de Tokenização**](/guides/features/tokenization). Com este SDK, os comerciantes que não são PCI podem salvar e tokenizar os cartões dos clientes incorporando um iframe em seu site/app.

## Obtenção dos cartões

Uma vez que temos os cartões dos nossos clientes salvos no Paylands, podemos recuperá-los e utilizá-los de diferentes formas. O primeiro passo será obter os cartões armazenados de um cliente.

A forma mais habitual de recuperar o cartão de um cliente é realizando uma chamada ao método da API do Paylands [para obter os cartões de um cliente](/reference/#operation/get-customer-cards). Através deste método obtém-se a lista de cartões armazenados no sistema para um dado `customer_ext_id`. O customer\_ext\_id deve coincidir com o que foi utilizado ao salvar o cartão no sistema ou ao criar a ordem de pagamento para o cliente.

```
curl --request GET 'https://api.paylands.com/v1/sandbox/customer/{customer_ext_id}/cards?unique=true' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \
```

Obteremos como resposta a lista de cartões do cliente com o `uuid` de cada um deles.

## Como usar o cartão

Uma vez que obtivemos os cartões de um cliente, podemos realizar pagamentos e autorizações de diferentes formas que são explicadas a seguir.

### Pagamento direto por webservice

O método de [**pagamento direto por webservice**](/reference/#operation/webservice-payment) permite realizar pagamentos sem a intervenção do usuário. Esta forma de pagamento é direcionada a comerciantes avançados que gerenciam suas próprias recorrências, já que são eles quem decidem a periodicidade das cobranças.

Para isso é necessário o uuid da ordem de pagamento `order_uuid` e o uuid de um cartão `card_uuid` previamente armazenado no sistema. Para gerar uma nova ordem podemos fazê-lo chamando o método [Gerar ordem de pagamento](/reference/#operation/generate-payment-order). A resposta nos retornará o `order_uuid`. Por segurança é solicitado o IP do usuário `customer_ip` que solicita o pagamento, podendo ser indicado em formato IPv4 ou IPv6.

```
curl --request POST 'https://api.paylands.com/v1/sandbox/payment/direct' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \--data-raw '{  "signature": "121149a0ba5361191d740fa898784a8b",  "order_uuid": "1F405EA3-9798-42A6-9E87-BD347EF67F55",  "card_uuid": "C10721E7-1404-45DC-8762-351DD9945D1D",  "customer_ip": "62.43.214.55"}'
```

### Pagamento tokenizado 3DS

Este método de **pagamento mediante o token de uma ordem** é similar ao [pagamento direto por webservice](#pagamento-direto-por-webservice), com a diferença de que neste caso ao usuário será mostrada a tela de [3DS](/guides/resources/glossary#3ds-2) antes de finalizá-lo.

Para gerar uma nova ordem podemos fazê-lo chamando o método [**Gerar ordem de pagamento**](/reference/#operation/generate-payment-order). A resposta nos retornará o `order_token`. Lançaremos um [**pagamento tokenizado 3DS**](/reference/#operation/3ds-tokenized) adicionando o `order_token` da ordem na URL da chamada.

### Autorização 3DS do cartão

Se para um cartão ainda não foi realizado um pagamento [3DS](/guides/resources/glossary#3ds-2) e queremos utilizá-lo mais adiante para lançar [pagamentos diretos por webservice](#pagamento-direto-por-webservice) sem querer cobrar ainda do usuário, é possível realizar uma **autorização 3DS do cartão lançando um pagamento 3DS de 0 euros**. Neste caso o usuário apenas se autentica de forma segura para mais adiante poder ser cobrado pelo comerciante sem que ele tenha que intervir novamente. Este é um processo habitual por exemplo em hotéis ou alguns provedores de assinaturas.

Deveremos gerar uma nova ordem de pagamento com o método [**Gerar ordem de pagamento**](/reference/#operation/generate-payment-order) indicando no campo `source_uuid` o uuid do cartão que desejamos autorizar e no campo `amount` indicar o valor `0`. Também é preciso indicar o campo `operative` com o valor `AUTHORIZATION`.

```
curl --request POST 'https://api.paylands.com/v1/sandbox/payment' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \--data-raw '{  "signature": "341f7de8e6fc49da8d8736473af6b03a",  "amount": 0,  "operative": "AUTHORIZATION",  "source_uuid": "UUID_TARJETA",  "secure": false,  "customer_ext_id": "test",  "service": "9A1BDCC8-DB30-4ED2-8523-62B330A67873",  "description": "Payment #1",  "additional": null,  "url_post": "https://my.website.com/process",  "url_ok": "https://my.website.com/ok",  "url_ko": "https://my.website.com/ko",  "template_uuid": "6412549E-933E-4DFE-A225-2E87FBF7623E",  "dcc_template_uuid": "BF418CA6-7043-4864-B36F-F02C2CF2B76B",  "save_card": true,  "reference": "50620",  "dynamic_descriptor": "PNP*NombreComercio",  "expires_in": 3600}'
```

## Cartões dinâmicos

Quando o usuário paga através de uma carteira digital como Google Pay ou Apple Pay, não está fornecendo o número do seu cartão mas um pseudônimo chamado DPAN (Device PAN). Este número tem a mesma aparência e está associado ao PAN (Primary Account Number) real do cartão, no entanto existem várias limitações em torno da informação que obtemos e seu uso. Estes meios de pagamento são conhecidos como **cartões dinâmicos** no Paylands.

|                    | PAN                                                         | DPAN                                                              |
| ------------------ | ----------------------------------------------------------- | ----------------------------------------------------------------- |
| **Definição**      | Identificador único por cartão                              | Identificador único por cartão e dispositivo                      |
| **Disponibilidade**| Todos os provedores o suportam, não requer uma carteira     | Apenas alguns provedores o suportam, requer uma carteira          |
| **Reutilização**   | Pode ser reutilizado em qualquer tipo de pagamento          | Só pode ser reutilizado em pagamentos sem a presença do usuário   |
| **Tokenização**    | O PAN é real e não pode ser invalidado                      | O DPAN é um alias e o usuário pode invalidá-lo posteriormente     |

Devido ao fato de que vários DPANs correspondem a um mesmo PAN, os cartões dinâmicos não contêm o campo `token` que é utilizado para determinar se dois cartões têm o mesmo PAN. Também devido ao fato de que o DPAN é um alias e não o número do cartão, não dispomos do `BIN`, embora sim do `last4` real do cartão porque o Google Pay e o Apple Pay o fornecem separadamente. Finalmente, a data de expiração não é fornecida, portanto não dispomos dela.

|                         | PAN | DPAN |
| ----------------------- | --- | ---- |
| UUID                    | ✔   | ✔    |
| Token único             | ✔   | ❌    |
| Data de expiração       | ✔   | ❌    |
| BIN                     | ✔   | ❌    |
| Last4                   | ✔   | ✔    |
| Holder/Alias\*          | ✔   | ✔    |
| Débito/Crédito          | ✔   | ✔    |
| País de emissão         | ✔   | ✔    |
| Nome do banco           | ✔   | ✔    |
| Pré-pago                | ✔   | ✔    |
| Descrição da marca      | ✔   | ✔    |

### Vantagens e desvantagens

A principal vantagem de solicitar os dados do cartão através do Google Pay ou Apple Pay é que em alguns casos o pagamento já é considerado autenticado e não é necessário realizar o 3DS. A desvantagem é que a autenticação é realizada pela carteira quando obtém os dados do cartão, portanto é necessário que o usuário interaja com a carteira cada vez. Isso impede que sejam lançados [**pagamentos tokenizados 3DS**](/reference/#operation/3ds-tokenized) com UUIDs de cartões dinâmicos.

Outra vantagem é que o usuário não precisa inserir os dados do seu cartão se já os tem registrados na sua carteira. Isso pode afetar positivamente a conversão já que para o usuário é mais cômodo selecionar seu cartão. Além disso, utilizar Google Pay ou Apple Pay pode ser mais familiar para certos usuários que desconfiam de inserir os dados do seu cartão em páginas web de terceiros.

Uma desvantagem é que, como desconhecemos o BIN do cartão, o usuário somente poderá ver os últimos 4 dígitos do seu cartão para poder identificá-lo. Isso é relevante se for mostrada ao usuário a lista de cartões que tem associados e precisa selecionar um deles.
