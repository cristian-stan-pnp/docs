When we talk about **tokenization**, we are referring to the action of **saving a payment method for future use**. When saving the payment method, a token is generated that will allow charges to be issued to the customer without needing to request their details again.

Tokenization protects sensitive data through a process of replacing the data with a non-sensitive equivalent, known as a token. The token has no external meaning or value. It is a reference, or identifier, that through a tokenization system, maps back to the sensitive data. This process securely collects sensitive credit card information and prevents data theft.

<Info>
  Although we usually call it "token" because we "tokenize" the card, we are referring to the card's UUID. The "token" field of a card is actually a value calculated from the card number, so two cards with the same [PAN](/guides/resources/glossary#pan) have different UUIDs but the same token. This token only serves to determine if two cards are the same. Whenever we want to use a card, it is necessary to provide its UUID.
</Info>

## How to save a card

We can store a card in Paylands using one of the following methods:

### Making a payment

We can tokenize any card when making a first payment with it. The usual method for merchants that are not [PCI](/guides/resources/glossary#pci-dss) compliant is to use the [**simple integration via payment form**](/guides/accept-payments/web-integration), although it is also possible to launch the payment form via:

* [Payment links](/guides/accept-payments/link-payments)
* [Moto payments](/guides/accept-payments/moto-payments)
* [E-commerce plugins](/guides/plugins/ecommerce).

In the notification received after the payment, an object of the processed `order` is included. Within the transaction there is also the `source` object which contains the `token` of the card used in the payment. This token will from now on be linked to the customer, so in new operations payments can be confirmed for this user without needing to request the card again.

These methods create an order with a [3DS](/guides/resources/glossary#3ds-2) transaction which generates a token with [SCA](/guides/resources/glossary#sca) allowing the merchant to send:

* [MITs](/guides/resources/glossary#mit)
* One-time charges (incremental)
* Recurring charges (subscriptions)

### Integrating the Tokenization SDK

In the case of building a platform, web or mobile, where you want the user to be able to save their card (like Netflix, Amazon) to later issue charges either for purchases or subscriptions, we have a [**Tokenization SDK**](/guides/features/tokenization) library. With this SDK, merchants that are not PCI compliant can save and tokenize customer cards by embedding an iframe in their web/app.

## Retrieving cards

Once we have our customers' cards saved in Paylands, we can retrieve them and use them in different ways. The first step will be to obtain a customer's stored cards.

The most common way to retrieve a customer's card is by making a call to the Paylands API method [to get a customer's cards](/reference/#operation/get-customer-cards). This method returns the list of cards stored in the system for a given `customer_ext_id`. The customer\_ext\_id must match the one used when saving the card in the system or when creating the payment order for the customer.

```
curl --request GET 'https://api.paylands.com/v1/sandbox/customer/{customer_ext_id}/cards?unique=true' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \
```

We will get as a response the list of the customer's cards with the `uuid` of each one.

## How to use the card

Once we have obtained a customer's cards, we can make payments and authorizations in different ways as explained below.

### Direct payment by webservice

The [**direct payment by webservice**](/reference/#operation/webservice-payment) method allows payments to be made without user intervention. This payment method is aimed at advanced merchants that handle their own recurrences, as they are the ones who decide the frequency of charges.

To do this, the payment order uuid `order_uuid` and the uuid of a card `card_uuid` previously stored in the system are needed. To generate a new order we can do so by calling the [Generate payment order](/reference/#operation/generate-payment-order) method. The response will return the `order_uuid`. For security purposes, the IP of the user `customer_ip` requesting the payment is required, which can be provided in IPv4 or IPv6 format.

```
curl --request POST 'https://api.paylands.com/v1/sandbox/payment/direct' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \--data-raw '{  "signature": "121149a0ba5361191d740fa898784a8b",  "order_uuid": "1F405EA3-9798-42A6-9E87-BD347EF67F55",  "card_uuid": "C10721E7-1404-45DC-8762-351DD9945D1D",  "customer_ip": "62.43.214.55"}'
```

### 3DS tokenized payment

This **payment method using the token of an order** is similar to the [direct payment by webservice](#direct-payment-by-webservice), with the difference that in this case the user will be shown the [3DS](/guides/resources/glossary#3ds-2) screen before completing it.

To generate a new order we can do so by calling the [**Generate payment order**](/reference/#operation/generate-payment-order) method. The response will return the `order_token`. We will launch a [**3DS tokenized payment**](/reference/#operation/3ds-tokenized) adding the `order_token` of the order in the call URL.

### 3DS card authorization

If a [3DS](/guides/resources/glossary#3ds-2) payment has not yet been made for a card and we want to use it later to launch [direct payments by webservice](#direct-payment-by-webservice) without wanting to charge the user yet, it is possible to perform a **3DS card authorization by launching a 3DS payment of 0 euros**. In this case the user only authenticates securely so that they can later be charged by the merchant without having to intervene again. This is a common process for example in hotels or some subscription providers.

We will need to generate a new payment order with the [**Generate payment order**](/reference/#operation/generate-payment-order) method, indicating in the `source_uuid` field the uuid of the card we wish to authorize and in the `amount` field the value `0`. It is also necessary to indicate the `operative` field with the value `AUTHORIZATION`.

```
curl --request POST 'https://api.paylands.com/v1/sandbox/payment' \--header 'Authorization: pk_test_3c140607778e1217f56ccb8b50540e00' \--header 'Content-Type: application/json' \--data-raw '{  "signature": "341f7de8e6fc49da8d8736473af6b03a",  "amount": 0,  "operative": "AUTHORIZATION",  "source_uuid": "UUID_TARJETA",  "secure": false,  "customer_ext_id": "test",  "service": "9A1BDCC8-DB30-4ED2-8523-62B330A67873",  "description": "Payment #1",  "additional": null,  "url_post": "https://my.website.com/process",  "url_ok": "https://my.website.com/ok",  "url_ko": "https://my.website.com/ko",  "template_uuid": "6412549E-933E-4DFE-A225-2E87FBF7623E",  "dcc_template_uuid": "BF418CA6-7043-4864-B36F-F02C2CF2B76B",  "save_card": true,  "reference": "50620",  "dynamic_descriptor": "PNP*NombreComercio",  "expires_in": 3600}'
```

## Dynamic cards

When the user pays through a wallet like Google Pay or Apple Pay, they are not providing their card number but a pseudonym called DPAN (Device PAN). This number has the same appearance and is associated with the real PAN (Primary Account Number) of the card, however there are several limitations regarding the information we obtain and its use. These payment methods are known as **dynamic cards** in Paylands.

|                    | PAN                                                         | DPAN                                                              |
| ------------------ | ----------------------------------------------------------- | ----------------------------------------------------------------- |
| **Definition**     | Unique identifier per card                                  | Unique identifier per card and device                             |
| **Availability**   | All providers support it, does not require a wallet         | Only some providers support it, requires a wallet                 |
| **Reusability**    | Can be reused in any type of payment                        | Can only be reused in payments without the user's presence        |
| **Tokenization**   | The PAN is real and cannot be invalidated                   | The DPAN is an alias and the user can invalidate it later         |

Because several DPANs correspond to the same PAN, dynamic cards do not contain the `token` field that is used to determine if two cards have the same PAN. Also because the DPAN is an alias and not the card number, we do not have the `BIN`, although we do have the real `last4` of the card because Google Pay and Apple Pay provide it separately. Finally, the expiration date is not provided, so we do not have it.

|                         | PAN | DPAN |
| ----------------------- | --- | ---- |
| UUID                    | ✔   | ✔    |
| Unique token            | ✔   | ❌    |
| Expiration date         | ✔   | ❌    |
| BIN                     | ✔   | ❌    |
| Last4                   | ✔   | ✔    |
| Holder/Alias\*          | ✔   | ✔    |
| Debit/Credit            | ✔   | ✔    |
| Country of issuance     | ✔   | ✔    |
| Bank name               | ✔   | ✔    |
| Prepaid                 | ✔   | ✔    |
| Brand description       | ✔   | ✔    |

### Advantages and disadvantages

The main advantage of requesting card data through Google Pay or Apple Pay is that in some cases the payment is already considered authenticated and it is not necessary to carry out 3DS. The disadvantage is that the authentication is carried out by the wallet when it obtains the card data, so the user needs to interact with the wallet each time. This prevents [**3DS tokenized payments**](/reference/#operation/3ds-tokenized) from being launched with dynamic card UUIDs.

Another advantage is that the user does not need to enter their card details if they already have them registered in their wallet. This can positively affect conversion since it is more convenient for the user to select their card. Additionally, using Google Pay or Apple Pay may be more familiar to certain users who are distrustful of entering their card details on third-party web pages.

A disadvantage is that because we do not know the BIN of the card, the user will only be able to see the last 4 digits of their card to identify it. This is relevant if the user is shown the list of cards they have associated and needs to select one of them.
